# TanStack Template Summary

This is the useful knowledge extracted from the original TanStack scaffold before cleaning it into a fresh starter.

## Core Architecture

TanStack Start is the application framework. It sits on top of Vite and TanStack Router, so the app is built around route files in `src/routes`.

The root route in `src/routes/__root.tsx` is the document shell. It controls HTML metadata, global stylesheet links, shared layout, scripts, and dev-only tools. Anything placed there wraps every route.

`src/routeTree.gen.ts` is generated by TanStack Router. It maps route files to typed route definitions. The file should not be hand-edited.

`src/router.tsx` creates the router instance. This is where shared router options, route context, preloading behavior, scroll restoration, and integrations are connected.

## Routing Lessons

Routes are file-based:

- `src/routes/index.tsx` becomes `/`.
- `src/routes/about.tsx` becomes `/about`.
- A route exports `Route = createFileRoute("/path")({ component })`.

Internal navigation should use `Link` from `@tanstack/react-router` so route targets stay type-safe and navigation remains client-side.

The root route uses `createRootRouteWithContext` so route loaders and components can access shared context, such as a React Query client.

## Data Loading

The template wired TanStack Query into the router through `@tanstack/react-router-ssr-query`. That gives the project a ready place for SSR-aware query usage without forcing every route to use it immediately.

For simple route data, TanStack Router loaders are also available. For reusable client/server cache behavior, TanStack Query is the stronger default.

## Styling And UI

Tailwind CSS 4 is configured through CSS imports and Vite, not a separate Tailwind config file.

The scaffold included shadcn-compatible primitives under `src/components/ui`. These are useful starter pieces because they depend on normal React, Tailwind classes, CSS variables, and the `cn` helper in `src/lib/utils.ts`.

Theme state is applied early with a small inline script in the root document to avoid a light/dark flash during hydration.

## Devtools

The scaffold included TanStack Devtools plugins for Router, Query, and demo Store state. The Store plugin was removed because it only supported demo code.

Router and Query devtools were kept in development mode because they are useful when building real routes and data flows.

## Removed Demo Surface

The scaffold had broad example code for AI chat, audio recording, text-to-speech, forms, tables, local collections, store state, guitar recommendations, Prisma, generated Prisma clients, and a SQLite demo database.

Those examples were useful for seeing how TanStack libraries can compose, but they created too much noise for a new app. Removing them leaves a smaller foundation with fewer dependencies and fewer false ownership signals.

## Fresh Start Rule

Keep framework wiring that real features will reuse:

- root document shell
- file-based routes
- router creation
- Query context
- shared components
- theme tokens
- formatting and test scripts

Remove feature demos until the application actually needs them:

- sample data
- generated clients
- demo assets
- provider-specific AI code
- throwaway state stores
- database schemas and seed files
